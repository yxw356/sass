## 1. 项目背景

### 1.1 背景

团队需要在**本地/内网**环境基于自有文档构建问答能力，要求：

- 数据不出域，文档与向量索引可控；
- 使用已部署的 **vLLM** 大模型服务，避免依赖公有云 LLM；
- 回答需能**追溯引用来源**，便于核对原文。

### 1.2 产品定位

KB Agent 是一套轻量级 **RAG（检索增强生成）** 服务：用户上传或粘贴文档内容 → 系统分块、向量化并存入本地知识库 → 用户提问时检索相关片段 → 调用 vLLM 生成带引用的回答。

### 1.3 目标用户

| 角色 | 诉求 |
|------|------|
| 业务/运营人员 | 通过网页上传文档、粘贴文字、自然语言提问 |
| 开发人员 | 通过 REST API 集成到内部系统 |
| 运维人员 | 本地部署、离线 Embedding、端口与配置清晰 |

### 1.4 非目标（当前版本不做）

- 多租户、权限体系、审计日志
- 云端托管（Supabase 等）— 仅作后续扩展方向
- 多轮对话记忆、会话管理
- 旧版 Word `.doc` 二进制格式（需用户转为 `.docx`）

## 2. 系统架构

### 2.1 逻辑架构

```
┌──────────────┐     ┌─────────────────────────────────────┐     ┌──────────────┐
│  浏览器/API  │────▶│  KB Agent (FastAPI :8080)           │────▶│  vLLM :8000  │
│              │     │  · 上传 / 文字入库 / 问答 / 健康检查  │     │  OpenAI 兼容 │
└──────────────┘     └──────────────┬──────────────────────┘     └──────────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
            data/uploads/    chroma_db/      models/bge-small-zh-v1.5/
            原始文件          向量索引         Embedding 模型
```

### 2.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| Web 框架 | FastAPI + Uvicorn |
| RAG 框架 | LlamaIndex |
| 向量库 | Chroma（本地持久化） |
| Embedding | HuggingFace `BAAI/bge-small-zh-v1.5`（本地） |
| LLM | vLLM（OpenAI 兼容 HTTP API） |
| 文档解析 | llama-index-readers-file、pypdf、python-docx、openpyxl 等 |

### 2.3 端口规划

| 端口 | 服务 | 说明 |
|------|------|------|
| 8080 | KB Agent | 业务 API + 网页 |
| 8000 | vLLM | 大模型推理（需单独启动） |

## 3. 功能需求

### 3.1 知识入库

#### FR-001 文件上传入库

| 项 | 说明 |
|----|------|
| 描述 | 用户上传文件，系统保存并解析为文本，分块后写入向量库 |
| 接口 | `POST /upload`（multipart，`file` 字段） |
| 支持格式 | `.txt`、`.md`、`.pdf`、`.docx`、`.xlsx`、`.xls`、`.csv`、`.pptx` |
| 存储 | 原文件保存至 `data/uploads/{uuid}_{原文件名}` |
| 成功响应 | `filename`、`message`、`documents_indexed` |
| 失败 | 400：格式不支持、文件名为空、解析无内容；500：索引失败 |

**版本**：v0.1.0 基础格式；v0.5.0 扩展 Office 等格式。

#### FR-002 文字直接入库

| 项 | 说明 |
|----|------|
| 描述 | 用户粘贴纯文本及可选标题，无需创建文件即可入库 |
| 接口 | `POST /ingest/text`，JSON：`content`（必填）、`title`（可选） |
| 存储 | 同步写入 `data/uploads/{uuid}_{标题}.txt` |
| 成功响应 | 与上传接口一致的 `UploadResponse` |

**版本**：v0.5.0。

#### FR-003 分块与向量化策略

| 项 | 说明 |
|----|------|
| 分块 | `SentenceSplitter`，`chunk_size=512`，`chunk_overlap=64` |
| 向量模型 | 可配置，默认 `BAAI/bge-small-zh-v1.5` |
| 离线 | 优先使用 `models/bge-small-zh-v1.5/`；支持 `EMBEDDING_LOCAL_FILES_ONLY` |

**版本**：v0.1.0；v0.3.0 离线增强。

---

### 3.2 知识问答

#### FR-004 检索问答

| 项 | 说明 |
|----|------|
| 描述 | 根据用户问题检索 Top-K 相关片段，交由 vLLM 生成回答 |
| 接口 | `POST /chat`，JSON：`message`（必填） |
| 检索 | `similarity_top_k` 可配置，默认 5 |
| 响应 | `answer`（纯正文）、`citations[]`（`index`、`source`、`excerpt`、`score`） |

**版本**：v0.1.0；v0.5.0 优化展示（`answer` 不再重复拼接引用块）。

#### FR-005 引用来源展示

| 项 | 说明 |
|----|------|
| 描述 | 用户可核对回答依据的文档名与摘录 |
| 网页 | 「参考来源」区：仅文件名 + 摘录，无 score/序号 |
| API | `citations` 保留完整字段供集成方使用 |
| 文件名 | 展示时去除上传 UUID 前缀 |

**版本**：v0.1.0 基础；v0.5.0 体验优化。

#### FR-006 vLLM 模型对接

| 项 | 说明 |
|----|------|
| 描述 | 通过 OpenAI 兼容接口调用本地 vLLM，模型名与部署一致 |
| 配置 | `VLLM_BASE_URL`、`VLLM_API_KEY`、`VLLM_MODEL`、`VLLM_CONTEXT_WINDOW` |
| 实现 | `VLLMOpenAI` 子类，跳过 OpenAI 官方模型名校验 |

**版本**：v0.1.0 基础；v0.3.0 修复校验问题。

### 3.4 运维与配置

#### FR-010 环境配置

通过 `.env` 或环境变量配置（见 `.env.example`）：

| 变量 | 说明 |
|------|------|
| `VLLM_*` | vLLM 连接与模型 |
| `CHROMA_PATH` | 向量库路径 |
| `UPLOAD_DIR` | 上传目录 |
| `EMBEDDING_MODEL` | Embedding 模型 ID 或本地路径 |
| `EMBEDDING_LOCAL_FILES_ONLY` | 是否仅使用本地模型文件 |
| `APP_HOST` / `APP_PORT` | 服务监听（文档示例用 uvicorn 参数） |

#### FR-011 Embedding 预下载

| 项 | 说明 |
|----|------|
| 脚本 | `python scripts/download_embedding_model.py` |
| 镜像 | 建议 `HF_ENDPOINT=https://hf-mirror.com` |
| 输出 | `models/bge-small-zh-v1.5/` |

**版本**：v0.3.0。

#### FR-012 清空知识库

| 项 | 说明 |
|----|------|
| 操作 | 停止服务 → 删除 `chroma_db/` 与 `data/uploads/` 内容 → 重启 |
| 说明 | 无单独 API，由运维手动执行 |

## 4. 非功能需求

### 4.1 部署环境

| 编号 | 要求 |
|------|------|
| NFR-001 | Python 3.10+ |
| NFR-002 | 可访问本地 vLLM（默认 `127.0.0.1:8000`） |
| NFR-003 | 磁盘：依赖 + Embedding 约 2GB+；向量库随文档增长 |
| NFR-004 | 支持内网/离线 Embedding（预下载模型） |

### 4.2 性能（参考目标，非硬性 SLA）

| 编号 | 说明 |
|------|------|
| NFR-005 | 首次问答/上传需加载 Embedding，耗时明显增加 |
| NFR-006 | 单次问答延迟主要取决于 vLLM 与文档规模 |

### 4.3 安全

| 编号 | 要求 |
|------|------|
| NFR-007 | `.env` 不入库；API Key 仅本地配置 |
| NFR-008 | 上传文件仅保存在服务端指定目录，无自动对外暴露 |

### 4.4 可维护性

| 编号 | 要求 |
|------|------|
| NFR-009 | 提供 CHANGELOG、需求文档、新手手册 |
| NFR-010 | 模块划分：`main`（API/UI）、`rag`（索引/问答）、`documents`（解析）、`config` |

## 8. 验收标准（v0.5.0）

### 8.1 部署验收

- [ ] `pip install -r requirements.txt` 成功
- [ ] Embedding 模型可加载（本地或镜像下载）
- [ ] vLLM 在 8000 端口可用，`VLLM_MODEL` 与 `/v1/models` 一致
- [ ] `curl /health` 返回 ok
- [ ] `http://127.0.0.1:8080/docs` 可打开且非白屏

### 8.2 功能验收

- [ ] 上传 `.txt` / `.pdf` / `.docx` / `.xlsx` 至少各一种成功，`documents_indexed >= 1`
- [ ] `POST /ingest/text` 成功，后续问答能检索到相关内容
- [ ] `POST /chat` 返回 `answer` 与 `citations`
- [ ] 网页问答区：正文与「参考来源」分区清晰，无重复大段引用、无 score 暴露

### 8.3 异常验收

- [ ] 上传不支持格式返回 400
- [ ] vLLM 未启动时 `/chat` 返回明确错误
- [ ] 空文字入库返回 400
